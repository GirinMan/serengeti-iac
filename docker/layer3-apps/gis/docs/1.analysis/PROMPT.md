# Role and Objective
당신은 뛰어난 역량을 가진 시니어 소프트웨어 아키텍트이자 GIS(지리정보시스템) 및 공간 데이터(Spatial Data) 처리 전문가입니다.
우리의 목표는 현재 심하게 엉켜있는(Spaghetti) 레거시 GIS 지하 시설물 시각화 서비스의 코드를 샅샅이 분석하여, **1) 기존 시스템의 복원 및 가동 방법을 찾고**, **2) 기존 구조 내에서의 데이터 확장 프로세스를 정의**하며, **3) 사내 표준 인프라(serengeti-iac)를 활용한 현대적 아키텍처로의 전면 재설계 청사진**을 도출하는 것입니다.

# Context & Pain Points
현재 레거시 시스템은 다음과 같은 치명적인 문제를 안고 있습니다:
1. 지역 확장성 불가: 특정 지역의 지도만 볼 수 있도록 코드가 강하게 결합(Tightly coupled)되어 있거나 하드코딩 되어 있습니다.
2. 데이터 확장성 불가: 새로운 방식의 측량 데이터나 신규 파이프라인/시설물 데이터를 삽입하고 매핑하기 매우 어려운 구조입니다.
3. 코드 가독성 최악: 비즈니스 로직, 데이터 접근(DB), 지도 렌더링 로직이 분리되지 않고 뒤섞여 있습니다.

# Context & Directory Structure
분석 대상이 되는 레거시 파일들은 Windows 데스크탑 환경에서 추출되었으며, 현재 `origin/` 폴더 하위에 다음과 같은 구조로 배치되어 있습니다. 이를 바탕으로 분석을 진행하십시오.
- `origin/movem/`: 기존 Windows User의 홈 디렉토리. 서버 시작 스크립트 및 관련 메모가 `movem/OneDrive/바탕 화면/` 경로 아래에 위치해 있습니다. (시스템 기동 방식 파악의 단서)
- `origin/pocheon/`: 서버 구동 소스 코드 및 관련 데이터가 모두 저장되어 있는 메인 분석 대상 디렉토리입니다.
- `origin/C_drive/`: 해당 데스크탑에서 서버가 실행되는 동안 생성/저장된 종속성 파일이나 런타임 파일들이 포함되어 있습니다.

# Tasks: Step-by-Step Analysis
제공된 레거시 코드베이스를 바탕으로 다음 단계에 따라 분석을 수행하고 보고서를 작성하십시오.

## Step 1: 기존 시스템 복원 및 아키텍처/의존성 파악 (Legacy Restoration & Inventory)
- `origin/movem/OneDrive/바탕 화면/`의 메모와 스크립트, 그리고 `origin/pocheon/`의 코드를 분석하여 **현재 상태 그대로 다시 서빙(가동)할 수 있는 구체적인 실행 방법**을 찾아내십시오.
- 현재 사용 중인 언어, 프레임워크, 주요 라이브러리(특히 지도 렌더링 및 공간 데이터 처리 관련)를 목록화하십시오.
- 시스템의 진입점(Entry point)과 핵심 모듈들의 역할을 리버스 엔지니어링하여 요약하십시오.
- 실행 환경 구성을 위한 의존성(`origin/C_drive/` 참고) 및 환경 변수 등을 문서화하십시오.

## Step 2: 공간 데이터 흐름 및 스키마 추출 (Data Flow & Schema)
- 기존 지하시설물 데이터(상하수도, 관로 등)가 데이터베이스에서 추출되어 클라이언트(지도)에 렌더링되기까지의 데이터 파이프라인을 추적하십시오.
- 데이터베이스 스키마 형태와 좌표계(CRS/EPSG) 변환 로직이 있다면 찾아내십시오.
- **[중요]** 측량 데이터가 입력되거나 렌더링되는 로직에서 강하게 결합된 병목 지점(Bottleneck)을 식별하십시오. 왜 새로운 데이터를 넣기 어려운지 기술적 원인을 분석하십시오.

## Step 3: 하드코딩 및 결합도 추적 (Hardcoding & Coupling)
- 특정 "지역"에 종속되게 만드는 하드코딩된 값(Bounding Box 좌표, 특정 행정동 ID, 정적 파일 경로 등)을 모두 찾아 파일명과 라인 넘버를 명시하십시오.
- 로직과 뷰(View)가 분리되지 않은 스파게티 지점을 찾아내고, 이를 어떻게 MVC 또는 API 기반 구조로 분리할 수 있을지 제안하십시오.

## Step 4: 기존 구조 기반의 데이터 확장 가이드 (Legacy Expansion Process)
- 만약 기존 시스템과 유사한 아키텍처를 유지한다고 가정할 때, **새로운 지역 지도(대한민국 한정)와 그에 해당하는 신규 지하시설물 데이터를 추가**하려면 어떤 프로세스를 거쳐야 하는지 정의하십시오.
- 데이터를 어떤 포맷(예: SHP, GeoJSON, 특정 DB 스키마 등)으로 준비하고, 코드나 DB의 어느 부분을 어떻게 수정해야 맵에 렌더링될 수 있는지 스텝별로 상세히 기술하십시오.

## Step 5: 현대적 아키텍처 전면 재설계 (Modernization Strategy)
- 기존 기능과 데이터를 100% 보존하면서 다음 요구사항을 만족하는 현대적 아키텍처를 설계하십시오.
  - 지역 확장이 자유로운 구조 (Dynamic Bounding Box & Spatial Indexing)
  - 다양한 형식의 측량 데이터(GeoJSON, 3D 좌표 등)를 쉽게 적재할 수 있는 유연한 파이프라인
  - 백엔드(데이터 제공 API)와 프론트엔드(GIS 렌더링)의 완벽한 분리
- **[인프라 제약 조건]** `/home/girinman/workspace/serengeti-iac`에 정의된 사내 데이터 인프라 레이어(예: PostgreSQL/PostGIS, Redis, Kafka, MinIO 등)를 최대한 활용하는 형태로 설계하십시오. 불가피하게 신규 인프라나 컴포넌트가 추가되어야 하는 경우에만 그 이유와 함께 명시하십시오.

# Output Format & Path Rules
당신의 분석 결과는 반드시 단일 답변으로 끝내지 말고, `docs/analysis/` 디렉토리를 생성하여 기능별/스펙별로 아래와 같이 마크다운 파일로 나누어 작성(저장)하십시오.

각각의 작업이 진행되기 전/후에 반드시 `docs/HISTORY.md`파일에 진척 상황을 기록하여 다음 세션 진행 시 확인 가능하게 하세요.

각 파일은 아래 항목을 반드시 포함하십시오:
- **AS-IS 분석 요약**: 현재 코드의 구조적 문제점 진단
- **Mermaid 다이어그램**: 해당 스펙과 관련된 흐름도 (Data Flow, 시스템 구조도 등)
- **핵심 개선 대상 리스트 (Action Items)**: 리팩토링 및 재개발 시 최우선으로 갈아엎어야 할 모듈과 이유
- **TO-BE 아키텍처 제안** (해당하는 파일에 한해): 최신 스택 기반 시스템 구조도 및 DB 스키마 개선 방향

저장 경로(예시):
1. `docs/analysis/01_legacy_restoration_guide.md` : Step 1의 기존 시스템 기동 방법 및 의존성 분석
2. `docs/analysis/02_data_flow_and_bottlenecks.md` : Step 2~3의 데이터 파이프라인 흐름도(Mermaid 포함) 및 하드코딩/결합도 분석 리스트
3. `docs/analysis/03_legacy_data_addition_process.md` : Step 4의 신규 지역(한국) 및 지하시설물 데이터 추가 프로세스와 데이터 포맷 정의
4. `docs/analysis/04_tobe_architecture_design.md` : Step 5의 `serengeti-iac` 기반 현대화 아키텍처 설계도(Mermaid 포함) 및 API/DB 스키마 개선 방향
